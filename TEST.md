## 3. 웹앱 관련 테스트

### 3.1 웹소켓 보안 인증서 우회 테스트

| 항목 | 내용 |
| --- | --- |
| 시나리오 | 로컬 PC에서 `python3 -m http.server 8000`으로 서빙된 웹앱(`blocks.html`)에 스마트폰/노트북 브라우저로 접속한 뒤, interpreter_node의 로봇 내부 IP(`wss://`)로 웹소켓 연결 시도 |
| 실제 결과 (1차) | ❌ 실패 — interpreter_node가 자체 서명(self-signed) 인증서로 `wss`를 서비스하여 브라우저가 보안 경고와 함께 연결을 차단함 |
| 조치 | 브라우저에서 로봇 IP(`https://[로봇IP]:포트`)로 먼저 직접 접속하여 인증서 예외(허용)를 수동 승인한 뒤 웹앱 재접속 |
| 재테스트 결과 | ✅ 통과 (예외 승인 후 정상 연결 확인) |

### 3.2 웹 서버 별도 기동 확인 테스트

| 항목 | 내용 |
| --- | --- |
| 시나리오 | `ros2 launch block_robot bringup.launch.py` 실행 직후, 런치 파일을 추가로 켜지 않고 바로 `http://[노트북_IP]:8000/blocks.html` 접속 시도 |
| 예상 결과(최초) | 런치 파일 하나로 ROS 2 노드와 웹 서버가 모두 기동되어 즉시 접속 가능 |
| 실제 결과 | ❌ 실패 — `bringup.launch.py`에는 웹 서버 실행이 포함되어 있지 않아 페이지가 로드되지 않음(연결 거부) |
| 원인 | `bringup.launch.py`는 interpreter_node, ultrasonic_node, buzzer_node 등 ROS 2 노드만 기동하며, `blocks.html`을 서빙하는 정적 HTTP 서버는 별도 프로세스로 분리되어 있음 |
| 조치 | 런치 파일 실행 후 `web/` 디렉토리에서 `python3 -m http.server 8000`을 별도 터미널로 추가 실행하도록 절차화하고 `README.md`에 단계로 명시 |
| 재테스트 결과 | ✅ 통과 (웹 서버를 수동으로 추가 실행한 뒤 정상 접속 확인) |

### 3.3 스피커(경고음) 자동재생 우회 테스트

| 항목 | 내용 |
| --- | --- |
| 시나리오 | 장애물 감지로 로봇이 정지할 때 브라우저 내장 경고음이 자동 재생되는지 확인 |
| 이슈 | 브라우저의 오디오 자동재생 정책으로 사용자 상호작용 이전에는 소리가 재생되지 않음 |
| 조치 | 실행 버튼 클릭(사용자 제스처) 시점에 Web Audio API 컨텍스트를 미리 활성화하여 이후 경고음이 즉시 재생되도록 우회 처리 |
| 재테스트 결과 | ✅ 통과 (실행 후 정지 상황에서 경고음 정상 재생 확인) |
